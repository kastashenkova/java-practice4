# Практика 4. Обробка анотацій
## Завдання
1. Створити свої звичайні класи та методи або використати наявні.
2. Створити власні анотації та впровадити їх в код з минулого кроку, щоб вони виконували задану вами логіку. Логіка повинна передбачати використання частини анотацій в runtime та частини під час компіляції.
3. Використати власні анотації для генерації нових класів через власний annotation processor.
4. Використати власні анотації для обробки і перевірки логіки в runtime(RetentionPolicy.RUNTIME). Код, що має ці анотації, повинен бути згенерований на кроці 3.

## Реалізація
**Тема проєкту:** робота з TCP протоколом, система моніторингу з чітким розділенням потоків (UI-сервер, сервер метрик) для забезпечення ізоляції даних під час мережевого обміну.

### Створені анотації
#### Анотації часу компіляції та генерації коду (SOURCE/CLASS)
- `@NetworkChannel` (@Target(ElementType.TYPE))

Використовується на етапі компіляції для позначення мережевих компонентів та вказує цільовий канал передачі даних (UI або METRICS). Кастомний процесор анотацій AnnotationProcessor знаходить ці класи та автоматично генерує новий клас маршрутизатора GeneratedSocketRouter.

- `@WarehouseAction` (@Target(ElementType.METHOD))

Маркерна анотація етапу компіляції, яка використовується для позначення методів бізнес-логіки обробки пакетів або запасів. Процесор аналізує ці методи, зчитує кількість їхніх аргументів та генерує клас GeneratedWarehouseExecutor.

- `@ThreadSafe` (@Target(ElementType.TYPE))

Маркерна анотація для статичного аналізу та документування. Вказує, що клас спроєктовано як потокобезпечний для роботи у конкурентному середовищі.

- `@VisibleForTesting` (@Target({ElementType.METHOD, ElementType.FIELD}))

Допоміжна анотація. Вказує, що метод або поле має розширений модифікатор доступу для можливості покриття їх юніт-тестами.

#### Анотації часу виконання (RUNTIME)

- `@ConnectionRules` (@Target(ElementType.METHOD))
  
Анотація для контролю мережевих обмежень (timeoutMs). Вона автоматично додається процесором до згенерованих (на кроці 3 описаного завдання) методів класу GeneratedSocketRouter. Під час виконання програми спеціальний валідатор зчитує цю анотацію через Reflection API та перевіряє, чи налаштування з'єднання відповідають вимогам безпечної передачі даних.

- `@ValidateParams` (@Target(ElementType.METHOD))

Анотація автоматично розміщується на методах щойно згенерованого класу GeneratedWarehouseExecutor. Вона зберігає мінімально необхідну кількість параметрів minParams, яку вимагав оригінальний метод. У runtime валідатор перевіряє вхідні масиви даних за допомогою рефлексії, щоб не допустити виклику методу з неповною кількістю аргументів.

- `@RetryOnFailure` (@Target(ElementType.METHOD))

Указує, що мережевий метод є нестабільним і його виклик слід повторити у разі помилки (IOException). Містить налаштування кількості спроб attempts та затримки між ними delayMs.

- `@LogExecutionTime` (@Target(ElementType.METHOD))

Використовується для профілювання. Указує, що час виконання методу необхідно заміряти та вивести в лог. Опціонально дозволяє логувати передані аргументи через параметр logArguments.
